# Sidecar IPC — JSON-RPC 2.0 method catalog

> The Deckhand Flutter UI spawns the Go sidecar (`deckhand-sidecar`) as a
> child process and talks to it via JSON-RPC 2.0 over stdin/stdout. One
> JSON object per line, UTF-8 encoded.

## Envelopes

### Request

```json
{"jsonrpc":"2.0","id":"<uuid>","method":"<method>","params":{...}}
```

- `id` is a UUID v4 string generated by the UI.
- `params` is always an object (never an array).

### Success response

```json
{"jsonrpc":"2.0","id":"<uuid>","result":{...}}
```

### Error response

```json
{"jsonrpc":"2.0","id":"<uuid>","error":{"code":<int>,"message":"<str>","data":{...}}}
```

Error code ranges:

| Range | Meaning |
|-------|---------|
| -32700…-32600 | JSON-RPC protocol errors (spec-reserved) |
| -32000…-32099 | Generic sidecar errors |
| -33000…-33099 | SSH errors (unused today; Dart handles SSH) |
| -34000…-34099 | Disk / flash errors |
| -35000…-35099 | Profile / fetch errors |
| -36000…-36099 | Network / upstream errors |

### Notifications (sidecar → UI only)

Sidecar-emitted progress messages — no `id`. All notifications include
`operation_id` in params so the UI can route the update to the request
that spawned it.

```json
{"jsonrpc":"2.0","method":"progress","params":{"operation_id":"<uuid>",...}}
```

## Methods

### Lifecycle

#### `ping`
Smoke test the sidecar is alive.
- **Params**: none
- **Result**: `{ sidecar_version: str, os: str, arch: str }`

#### `version.compat`
Check whether the sidecar speaks the UI's protocol version.
- **Params**: `{ ui_version: str }`
- **Result**: `{ compatible: bool, sidecar_version: str }`

#### `host.info`
Return host platform info + the OS-appropriate data directories.
- **Params**: none
- **Result**:
  ```json
  { "os":"windows|darwin|linux",
    "arch":"amd64|arm64",
    "home_dir":"...", "cache_dir":"...", "data_dir":"...",
    "settings_file":"..." }
  ```

#### `shutdown`
Ask the sidecar to exit cleanly.
- **Params**: none
- **Result**: `{ ok: true }` (sidecar exits immediately after responding)

---

### Disks

#### `disks.list`
Enumerate local disks.
- **Params**: none
- **Result**: `{ disks: DiskInfo[] }`
- **`DiskInfo`**:
  ```json
  { "id":"PhysicalDrive3",
    "path":"\\\\.\\PHYSICALDRIVE3",
    "size_bytes": 7818182656,
    "bus":"USB",
    "model":"Generic STORAGE DEVICE",
    "removable": true,
    "partitions":[{"index":1,"filesystem":"...","size_bytes":..,"mountpoint":"F:\\"}] }
  ```

#### `disks.hash`
SHA-256 of a local file.
- **Params**: `{ path: str }`
- **Result**: `{ sha256: hex, path: str }`

#### `disks.read_image`
dd a disk into a file (for backups). Emits `progress` notifications.
- **Params**: `{ device_id: str, path?: str, output: str }`
- **Result**: `{ sha256: hex, output: str }`
- **Notifications**: `progress` with `{ bytes_done, bytes_total, phase: "reading"|"done" }`
- **Note**: Requires read access to raw block devices. On Windows this
  usually means running elevated; on Linux the user must be in the disk
  group or root.

#### `disks.write_image`
Write an image to a disk. In the main sidecar this **always returns an
error** — the UI must dispatch via the elevated helper binary
(`deckhand-elevated-helper`) which performs the actual write.
- **Params**: `{ image_path: str, disk_id: str, confirmation_token: str }`
- **Result**: — (always errors with "elevation required")

---

### OS image download

#### `os.download`
HTTP GET a large file with progress + optional sha256 verification.
- **Params**: `{ url: str, dest: str, sha256?: hex }`
- **Result**: `{ sha256: hex, path: str }`
- **Notifications**: `progress` with `{ bytes_done, bytes_total, phase: "downloading"|"done" }`

---

### Profiles

#### `profiles.fetch`
Shallow-clone (`depth=1`) a profile repo at a ref via go-git.
- **Params**: `{ repo_url: str, ref: str, dest: str, force?: bool }`
- **Result**:
  ```json
  { "local_path":"...",
    "resolved_sha":"...",
    "resolved_ref":"...",
    "was_cached": bool }
  ```

## Elevated helper (separate binary)

`deckhand-elevated-helper` is **not** a JSON-RPC peer — it's a single-op
Go binary the UI launches with platform-native elevation (UAC /
AuthorizationExecuteWithPrivileges / pkexec).

### Invocation

```
deckhand-elevated-helper write-image \
  --image <path> --target <disk_id> --token <confirmation_token> \
  [--verify] [--sha256 <hex>]
```

### Output

Line-delimited JSON on stdout:

```json
{"event":"preparing","device":"\\\\.\\PHYSICALDRIVE3","image":"..."}
{"event":"progress","phase":"writing","bytes_done":..,"bytes_total":..}
{"event":"progress","phase":"write-complete","sha256":"...","bytes_done":..,"bytes_total":..}
{"event":"progress","phase":"verified","sha256":"..."}
{"event":"done","sha256":"...","bytes":..}
```

On failure: `{"event":"error","message":"..."}` then exit non-zero.

## Framing implementation notes

- Each JSON message is on a single line; messages are split on `\n`.
- Max line length: 16 MiB. Longer messages (which shouldn't happen —
  binary payloads go through file paths, not JSON) will be rejected by
  the sidecar scanner.
- Responses for different requests may arrive out of order; the UI
  correlates via `id`.
- Handlers run in their own goroutines — multiple long-running ops can
  progress concurrently.
